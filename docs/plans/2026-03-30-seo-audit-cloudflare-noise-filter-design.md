# SEO Audit Cloudflare Noise Filter Design

## Why this change exists

The latest live SEO audit has dropped from six issues to one. The remaining issue is not a site-quality problem:

- `https://cloudcostkit.com/cdn-cgi/l/email-protection/`

This URL is generated by Cloudflare email protection, not by the site's content strategy. Treating it as a normal
internal page creates a false positive and makes the audit noisier than it should be.

## Scope of this change

This change only affects the local SEO audit tooling:

- `scripts/seo-audit.mjs`
- a small helper module for link extraction logic
- test coverage for ignoring Cloudflare internal protection paths

It does not change the live site's public content or routing.

## Problem pattern

The audit currently extracts same-site links from rendered HTML and treats them as crawl candidates. That is generally
correct, but it is too literal for internal platform paths such as:

- `/cdn-cgi/l/email-protection`

Those links are infrastructure behavior, not editorial routes. Including them in the crawl causes a false `non200`
issue.

## Options considered

### Option 1: Leave the audit noisy and remember to ignore the issue manually

Pros:

- no code changes

Cons:

- keeps the report less trustworthy
- wastes attention on a non-content problem every time the audit runs

### Option 2: Filter Cloudflare internal paths during internal-link extraction

Pros:

- minimal fix
- aligns the audit with the site's real editorial surface
- easy to cover with a focused test

Cons:

- adds one small rule to the crawler

### Option 3: Broadly ignore all non-standard internal links

Pros:

- could suppress more noise

Cons:

- higher chance of hiding legitimate issues
- unnecessary for the current evidence

## Recommended approach

Choose Option 2.

The audit should ignore only known infrastructure-only paths, starting with `/cdn-cgi/`. This is precise enough to fix
the current false positive without weakening the rest of the crawler.

## Final design

### Link extraction helper

Extract the URL-normalization and internal-link extraction logic from `scripts/seo-audit.mjs` into a small helper
module. That keeps the behavior testable without importing the full audit script.

### Filter rule

If a normalized same-site URL has a pathname starting with `/cdn-cgi/`, do not include it in the extracted internal
link set.

This rule should apply before the crawl queue is built so the audit never treats the link as a real route candidate.

### Test coverage

Add one focused Node test that proves:

- normal internal links are still extracted
- `mailto:` and fragment links are still ignored
- Cloudflare `/cdn-cgi/l/email-protection` links are ignored

## Success criteria

This change is successful when:

- the SEO audit no longer reports `/cdn-cgi/l/email-protection/` as a site issue
- normal internal link extraction behavior remains intact
- the repo still passes `npm test`
- a fresh `npm run audit:seo` report drops from `1` issue to `0`
